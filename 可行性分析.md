# Java 后端改造可行性分析

> 项目：2025 Blog（Next.js 前端 + GitHub API CMS）→ 改造为 Next.js 前端 + Java 后端

---

## 一、现状分析

### 1.1 当前架构

```
┌──────────────────────────────────────────────────┐
│                   Next.js 前端                     │
│  (React 19 + TypeScript + Tailwind CSS + Motion)  │
└────────┬─────────────────────────────────────────┘
         │ ① JWT (RSA-SHA256) 签名
         │ ② GitHub REST API (CRUD)
         ▼
┌──────────────────────────────────────────────────┐
│              GitHub 私有仓库                        │
│  • public/blogs/{slug}/index.md   (博客正文)       │
│  • public/blogs/{slug}/config.json (博客元信息)     │
│  • public/blogs/index.json        (博客索引)        │
│  • public/blogs/categories.json   (分类列表)        │
│  • src/config/site-content.json   (站点配置)       │
│  • src/config/card-styles.json    (卡片布局)       │
│  • src/app/{模块}/list.json       (各模块数据)      │
│  • public/images/**               (图片资源)       │
└────────┬─────────────────────────────────────────┘
         │ ③ Webhook 触发
         ▼
┌──────────────────────────────────────────────────┐
│          Vercel / Cloudflare Pages                │
│              自动重新部署                           │
└──────────────────────────────────────────────────┘
```

**核心特点：**
- 前端直接通过 GitHub API 读写仓库文件实现 CMS
- 使用 GitHub App + JWT (RSA-SHA256) 进行身份认证
- 内容以 JSON/Markdown 文件形式存储于仓库中
- 部署平台通过 Git 变更自动触发重新构建

### 1.2 当前数据流（以"发布博客"为例）

```
前端 → getAuthToken() → getRef(分支最新SHA) → createBlob(上传图片/md/json)
     → createTree(构建文件树) → createCommit(创建提交) → updateRef(更新分支引用)
     → 等待Vercel部署 → 内容生效
```

### 1.3 涉及的数据实体与存储位置

| 实体 | 当前存储位置 | 数据格式 |
|------|-------------|---------|
| 博客文章 | `public/blogs/{slug}/index.md` + `config.json` | Markdown + JSON |
| 博客索引 | `public/blogs/index.json` | JSON 数组 |
| 分类列表 | `public/blogs/categories.json` | JSON 数组 |
| 站点配置 | `src/config/site-content.json` | JSON 对象 |
| 卡片布局 | `src/config/card-styles.json` | JSON 对象 |
| 关于页面 | `src/app/about/list.json` | JSON 对象 |
| 博主收藏 | `src/app/bloggers/list.json` | JSON 数组 |
| 项目展示 | `src/app/projects/list.json` | JSON 数组 |
| 分享链接 | `src/app/share/list.json` | JSON 数组 |
| 图片画廊 | `src/app/pictures/list.json` | JSON 数组 |
| 代码片段 | `src/app/snippets/list.json` | JSON 数组 |
| 图片资源 | `public/images/**` + `public/blogs/{slug}/*` | 二进制文件 |
| 点赞数据 | Cloudflare Worker (`blog-liker.yysuni1001.workers.dev`) | KV 存储 |

## 二、改造目标

将 **前端 → GitHub API** 的数据写入链路，替换为 **前端 → Java REST API → 数据库/对象存储**。

```
改造后架构：

┌──────────────────────┐     HTTP REST      ┌──────────────────────┐
│    Next.js 前端       │ ◄─────────────────► │    Java Spring Boot   │
│  (保留全部UI和样式)     │    JSON + 文件上传   │    后端服务            │
└──────────────────────┘                    └────────┬─────────────┘
                                                     │
                                              ┌──────▼─────────────┐
                                              │   MySQL / PostgreSQL │
                                              │   (结构化数据)        │
                                              ├──────────────────────┤
                                              │   对象存储 (OSS/MinIO) │
                                              │   (图片/文件)         │
                                              └──────────────────────┘
```

## 三、可行性评估：✅ 完全可行

### 3.1 前端改动范围

**好消息：前端改动极小，约 90% 的代码可以保持不变。**

#### 不需要改的部分（直接复用）

| 层级 | 内容 | 原因 |
|------|------|------|
| **所有页面组件** | 首页、博客列表/详情、关于、博主、项目、分享、图片、时钟、Live2D、工具等 | 这些组件通过 Hooks 获取数据，不直接依赖 GitHub API |
| **所有 UI 组件** | Card、DialogModal、CodeBlock、LikeButton、ColorPicker、MarkdownImage 等 | 纯展示/交互组件，无外部依赖 |
| **样式系统** | Tailwind CSS 配置、CSS 变量、动画 | 完全不涉及后端 |
| **状态管理** | Zustand Store（config-store, layout-edit-store, write-store 等） | 只需修改 Store 中数据的获取方式 |
| **Markdown 渲染** | marked + Shiki + KaTeX 渲染管线 | 纯前端逻辑 |

#### 需要改的部分（约 10%）

| 模块 | 当前实现 | 改造方式 | 工作量 |
|------|---------|---------|--------|
| **`src/lib/github-client.ts`** | GitHub REST API 封装 | 替换为 Java 后端 API 封装 | 中 |
| **`src/lib/auth.ts`** | GitHub App JWT 认证 | 替换为 JWT 登录认证（用户名/密码 → JWT Token） | 中 |
| **`src/hooks/use-auth.ts`** | GitHub 密钥管理 | 替换为登录状态管理 | 小 |
| **所有 `services/push-*.ts`** | `pushSiteContent`, `pushBlog`, `pushBloggers` 等 8 个文件 | 改为调用 Java REST API | 中 |
| **`src/lib/blog-index.ts`** | GitHub 上读写 index.json | 改为调用 API 或直接由后端维护 | 小 |
| **`src/lib/load-blog.ts`** | `fetch(/blogs/...)` 加载文件 | **可以保持不变**（静态文件仍可从 Next.js public 目录提供）或改为 API 调用 | 极小 |
| **`src/app/sitemap.ts` / `rss.xml`** | 读取本地 JSON 文件 | 改为从 API 获取（SSR 时调用后端） | 小 |
| **`src/consts.ts`** | `GITHUB_CONFIG` 环境变量 | 替换为 `API_BASE_URL` 等后端配置 | 极小 |
| **`src/app/(home)/config-dialog`** | 密钥导入 → 保存 | 改为登录 → 保存 | 小 |

### 3.2 前端改造的核心模式

所有 push service 目前都遵循相同的模式，改造时只需要换掉这一层即可：

```typescript
// 改造前：通过 GitHub API 写入仓库
async function pushBlog(params) {
  const token = await getAuthToken()           // GitHub JWT
  const ref = await getRef(token, ...)         // GitHub API
  const blob = await createBlob(token, ...)    // GitHub API
  const tree = await createTree(token, ...)    // GitHub API
  const commit = await createCommit(token, ...) // GitHub API
  await updateRef(token, ...)                  // GitHub API
}

// 改造后：通过 REST API 写入后端
async function pushBlog(params) {
  const token = await getAuthToken()           // 自己的 JWT
  const formData = new FormData()
  formData.append('config', JSON.stringify(config))
  formData.append('markdown', md)
  formData.append('cover', coverFile)
  formData.append('images', imageFiles)
  await fetch(`${API_BASE}/api/blogs`, {       // 一个请求搞定
    method: 'POST',
    headers: { Authorization: `Bearer ${token}` },
    body: formData
  })
}
```

### 4.4 认证方案对比

| 维度 | 改造前 (GitHub App) | 改造后 (Java 后端) |
|------|-------------------|-------------------|
| 认证方式 | 导入 PEM 私钥文件 | 用户名 + 密码登录 |
| Token | RSA-SHA256 签名 JWT | HMAC-SHA256 签名 JWT |
| 存储 | IndexedDB (可选缓存) | localStorage / Cookie |
| 权限模型 | GitHub App 仓库权限 | RBAC（单用户 Admin） |
| 安全性 | 依赖私钥保管 | 依赖密码强度 + Token 过期 |

**推荐方案：**
- 首次部署时生成默认管理员账号密码
- 使用 Spring Security + JWT 无状态认证
- Token 有效期 24 小时，支持 refresh token
- 点赞接口无需认证（基于 IP 限流）

### 4.5 图片存储方案

| 方案 | 优点 | 缺点 | 推荐度 |
|------|------|------|--------|
| **MinIO 自建** | 兼容 S3 API，开源免费，部署简单 | 需要额外维护 | ⭐⭐⭐⭐⭐ |
| 本地文件系统 | 零依赖，开发简单 | 不便于扩展和备份 | ⭐⭐⭐ |
| 阿里云 OSS | 稳定可靠，CDN 加速 | 需要付费 | ⭐⭐⭐⭐ |
| 存储到数据库 | 与业务数据统一管理 | 性能差，不推荐 | ⭐ |

**推荐：MinIO**（用 Docker 部署，兼容 S3 协议，后续可无缝迁移到云 OSS）

如果希望保留现有博客内容，需要编写一次性迁移脚本：

```java
// 迁移脚本思路
1. 克隆或下载 GitHub 仓库到本地
2. 读取 public/blogs/index.json → 遍历每篇文章
3. 读取 public/blogs/{slug}/config.json + index.md
4. 逐条写入数据库

6. 更新图片 URL 映射
```

## 六、风险评估

### 6.1 技术风险

| 风险 | 等级 | 说明 | 缓解措施 |
|------|------|------|---------|
| **SEO 影响** | 中 | 博客内容从静态文件变为动态 API，影响搜索引擎收录 | 使用 Next.js SSR/SSG 预渲染博客页面 |
| **首屏加载** | 低 | API 调用可能比直接读文件慢 | 首页数据 SSR 预取；使用 SWR 缓存 |
| **Markdown 图片路径** | 低 | 图片从 `/blogs/slug/xxx.webp` 变为对象存储 URL | 后端返回时处理 URL 映射 |
| **并发写入** | 低 | 多端同时编辑可能冲突 | 单用户场景影响小；可加乐观锁 |
| **与上游项目同步** | 中 | 原项目持续更新，改造后难以合并上游 | Fork 后独立维护；或抽象 API 层减少差异 |

## 八、结论与建议

### 结论：✅ 完全可行，且收益显著

将 GitHub API CMS 替换为 Java 后端是**低风险、高收益**的改造：

1. **前端改动量小**（约 10%），所有页面样式和交互逻辑完全保留
2. **后端需求清晰**，数据模型和 API 契约已有明确定义
3. **用户体验提升明显**：发布文章无需等待 GitHub 部署（3分钟 → 即时生效）
4. **扩展性好**：后续可以轻松添加评论系统、全文搜索、访问统计等功能
5. **学习价值高**：涵盖 Spring Boot、JWT、文件上传、RESTful API 等核心知识点

### 建议

1. **保留 Next.js 作为前端 + BFF 层**：利用 Next.js 的 SSR/SSG 能力，博客详情页可以在构建时或请求时从 Java 后端获取数据并预渲染，保证 SEO
2. **图片仍然放在 Next.js 的 `/public` 目录或使用 MinIO**：建议用 MinIO 统一管理，前端通过 URL 直接访问
3. **点赞功能直接并入 Java 后端**：不再需要单独的 Cloudflare Worker
4. **数据库选 MySQL 8.x**：生态最成熟，学习资源最多
5. **先用 Docker Compose 本地开发**：一键启动 MySQL + MinIO + Java 后端 + Next.js 前端
